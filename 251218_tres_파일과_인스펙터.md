### 🧠 배운 개념 정리 (핵심)

#### 1. `.tres` 파일과 Resource의 역할

* `.tres`는 **텍스트 기반 Resource 에셋**
* Unity의 `.asset` (ScriptableObject 인스턴스)와 거의 동일한 위치
* 특징:

  * 코드와 데이터 명확히 분리
  * Inspector에서 직접 값 수정 가능
  * 여러 Node / Scene에서 공유 참조 가능

👉 감각적으로는
**“SO를 하나 더 일반화한 개념”**에 가까움

---

#### 2. Inspector 제어 감각 비교

**Unity**

* `[SerializeField]`
* `[CreateAssetMenu]`
* 컴파일은 자동
* Inspector 반영이 즉각적

**Godot (C#)**

* `[Export]` 속성으로 Inspector 노출
* Resource 스크립트를 작성한 뒤
* **명시적으로 C# 빌드 필요**

👉 여기서 한 번 흐름이 끊김

---

### 🧪 오늘 막혔던 지점 (중요 ⭐)

#### 🔥 C# 스크립트 빌드 흐름

* Resource 스크립트를 작성했는데
* Inspector에 바로 반영되지 않음
* 이유:

  * Godot C#은 **편집기와 런타임 사이에 빌드 단계가 존재**
  * Unity처럼 “뒤에서 알아서 컴파일”해주지 않음

👉 해결의 실마리:

* C# 프로젝트 빌드 개념을 **명확히 의식해야 함**
* Godot은 “스크립트 = 즉시 반영”이 아니라
  **“스크립트 → 빌드 → 에디터 반영” 구조**

이 지점에서

> “아, 고도는 C#을 진짜 ‘외부 언어’로 대하는구나”
> 라는 느낌을 받음

---

### 🔁 Unity와 비교하며 느낀 차이

| 항목           | Unity            | Godot            |
| ------------ | ---------------- | ---------------- |
| 데이터 에셋       | ScriptableObject | Resource (.tres) |
| Inspector 노출 | 즉시               | 빌드 후 반영          |
| C# 처리        | 엔진 내부 통합         | .NET 프로젝트 기반     |
| 감각           | 매끄럽다             | 명시적이다            |

---

### 💡 오늘의 통찰

* Godot은 편의성보다 **구조적 명확성**을 우선하는 엔진
* C#을 쓰더라도 “엔진에 녹아든 언어”가 아니라
  **외부 언어를 연결해서 쓰는 느낌**
* 이 빌드 단계가 처음엔 피곤하지만,
  장기적으로는 **무엇이 언제 반영되는지 예측 가능**해질 것 같음
